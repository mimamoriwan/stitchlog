# Stitchlog — 仕様書

クロスステッチャーが世界中で図案を作り、刺し、繋がるプラットフォーム

発想の切り口：**切り口6「他にも使えるようにする」× 切り口25「自分でうまく繋がってくれるようにする」**

最終更新：2026年5月

---

## 1. プロジェクト概要

### 一言で表すと
「クロスステッチ版のnote + Etsy + 進捗管理アプリ」を一つに統合したプラットフォーム。

### 解決する根本問題
- 既存の図案は種類が限られる（風景・キャラクターが中心）
- 著作権の問題で愛犬・家族など個人的なモチーフを刺せない
- 完成作品の発表場所がSNSしかなく、図案作者にフィードバックが届かない
- 制作中の気づきや注意点が個人の中に埋もれる
- 日本語×日本糸ブランド対応の包括的サービスが存在しない

### コアバリュー
**「作る人と刺す人が固定されない経済圏」**

全ユーザーが図案作者にも刺し手にもなれる。ポイント経済と展示室機能で、図案の価値が時間と共に蓄積される。

## 2. ターゲットユーザー

### コア層
- クロスステッチ歴3年以上の経験者
- 25〜55歳、女性中心、世界中に分散
- 推定リーチ可能母数：日本15万人、世界数百万人

### ペルソナ
1. **作り手志向**：自分のデザインを世に出したい
2. **記録志向**：制作経験を残し共有したい
3. **個人モチーフ志向**：愛犬・家族など固有の対象を刺したい
4. **発見志向**：新しい図案を能動的に探したい

### 国別優先順位
- **第1波**：日本・英語圏（米英豪）
- **第2波**：ドイツ・フランス・北欧
- **第3波**：ロシア・スペイン語圏・台湾

## 3. コア機能（全体像）

### 3.1 図案作成
- ユーザー写真をAIで図案化
- 色数・サイズ・糸ブランドを選択
- 細部調整（シャープ化、コントラスト、特定領域強調）
- DMC・Anchor・Olympus・Cosmoの糸番号対応
- 糸ブランド間の色変換マッピング

### 3.2 制作ビューアー
- アプリ内で図案を見ながら刺す
- マス目タップで進捗マーキング
- ズーム・パン・現在位置ハイライト
- 「ここまで刺した」写真記録
- 中断後の復帰位置自動表示

### 3.3 制作メモ
- 図案上の特定位置に紐づくメモ
- 「13列目：色替え注意」のような知見記録
- 自分用と公開用を切り替え可能
- 写真添付可能

### 3.4 公開・販売
- 図案+完成写真+制作メモをセットで投稿
- 価格設定（無料公開/有料販売）
- 著作権確認チェックフロー
- 図案＋完成写真がセットでないと公開不可（品質担保）

### 3.5 展示室
- 図案ごとに購入者の完成作品が集まる
- 国別表示（国旗の並び）、完成までの時間統計
- 作者プロフィールページ
- 累計購入数・完成数・評価が可視化

### 3.6 ポイント経済
- 図案が売れる→作者に売上の一部がポイントとして還元（**制作ポイント**）
- 完成時に購入者から作者へ感謝を表現できる（**感謝ポイント**）
- ポイントは他の図案購入に使える
- **現金換金は不可**（資金決済法回避）

### 3.7 検索・発見
- モチーフカテゴリ（動物・植物・人物・風景・抽象・和柄）
- 難易度・所要時間・色数フィルター
- 「あなたと似た作品を作っている人」レコメンド
- 季節・トレンド特集

## 4. 既存サービスとの差別化

| サービス | 機能 | Stitchlogとの差 |
|---------|------|----------------|
| Love It Stitch It | 変換+マーケット | 制作メモ・展示室・ポイント経済なし |
| Stitchly | 変換+進捗管理 | コミュニティ・マーケットなし |
| Xstitchify | 変換特化 | コミュニティ機能ゼロ |
| Etsy | 図案販売 | クロスステッチ特化UXなし、完成品ループなし |
| Pinterest | 図案発見 | 売買・進捗管理なし |
| Instagram | 完成品共有 | 図案と作品が紐づかない |

### 決定的優位点
1. **作る/刺す/発表/再利用が一つのループに**
2. **ポイント経済圏でユーザーが離れにくい**
3. **展示室が図案の価値を可視化**
4. **多言語×多糸ブランド対応で世界市場対応**
5. **制作メモが知識ネットワークを形成**
6. **時間効果＋ネットワーク効果のハイブリッド**

## 5. プラットフォーム戦略

### 5.1 ターゲット環境
- **PWA**：全世界、初期導線、低摩擦体験
- **iOS / iPadOS ネイティブ**：深い使用体験、Apple Pencil活用
- **macOS ネイティブ**：将来的（Phase 5以降）

### 5.2 各プラットフォームの役割分担

| 機能 | PWA | iOS | iPadOS | macOS |
|------|-----|-----|--------|-------|
| 図案作成 | ◎ | ○ | ◎ | △ |
| ビューアー（刺すとき）| △ | ○ | ◎ | × |
| 公開・販売 | ◎ | ◎ | ◎ | △ |
| 展示室閲覧 | ◎ | ○ | ◎ | ◎ |
| Apple Pencil編集 | × | × | ◎ | × |
| ローカルAI処理 | △ | ◎ | ◎ | ◎ |
| 高度な図案編集 | △ | × | ○ | ◎ |

### 5.3 戦略の核心
- **PWAは「すべての機能の入口」**：登録〜初回変換までの摩擦をゼロに
- **ネイティブは「特化体験」**：刺す時のiPadが理想体験
- **共通バックエンド**でデータ・購入履歴・ポイントを一元管理
- App Store手数料を回避するため、課金導線はPWA優先

## 6. 技術アーキテクチャ

### 6.1 全体構造

```
┌─────────────────────────────────┐
│   フロントエンド（複数）         │
│  PWA   /   iOS   /   iPadOS    │
└─────────────────────────────────┘
              ↓ ↑ REST/tRPC
┌─────────────────────────────────┐
│   バックエンドAPI（共通）        │
│   認証・データ・決済・通知       │
└─────────────────────────────────┘
              ↓ ↑
┌─────────────────────────────────┐
│   データレイヤー                 │
│   Postgres / S3 / Redis          │
└─────────────────────────────────┘
```

### 6.2 推奨技術スタック

**バックエンド**
- Node.js + TypeScript（型をフロントと共有）
- フレームワーク：Hono or Next.js APIルート
- データベース：PostgreSQL（Supabase推奨で初期速度確保）
- ストレージ：Cloudflare R2 or AWS S3
- 画像処理：sharp + 独自AIパイプライン
- 認証：Clerk or Supabase Auth
- 決済：Stripe（国際対応必須）
- 通知：OneSignal or Firebase Cloud Messaging

**PWA**
- Next.js 14+ App Router
- Tailwind CSS + shadcn/ui
- TanStack Query
- Service Worker（オフライン対応）
- WebAssembly（クライアント側画像処理の余地）

**ネイティブ**
- SwiftUI
- The Composable Architecture（状態管理）
- Core ML / MLX（ローカルAI処理）
- PencilKit（Apple Pencil統合）
- 独自バックエンドで全プラットフォーム統一

### 6.3 AI処理の分散戦略
- Phase 1：サーバー側で処理（OpenAI/Replicate/独自モデル）
- Phase 3：ネイティブ版はローカル処理（Core ML）
- PWAはサーバー処理を維持、ネイティブは料金プランで差別化可能

## 8. ポイント経済圏設計

### 8.1 ポイントの種類

| 種類 | 獲得方法 | 用途 |
|------|---------|------|
| 制作ポイント | 自分の図案が売れた時に売上の30%相当を還元 | 他の図案購入に使用可 |
| 感謝ポイント | 完成時に他ユーザーから贈呈される | 同上 |
| 初回ボーナス | 登録時付与 | 同上 |

### 8.2 ポイントの流れ
```
A が B の図案（500円）を購入
  ↓
B に制作ポイント150pt（30%相当）が付与
A の支払いは現金 or ポイント
  ↓
A が完成して B に感謝ポイント50pt贈呈
  ↓
B は累計200ptを別の作者の図案購入に使用
```

### 8.3 法的考慮
**現金換金は不可**とすることで、資金決済法の前払式支払手段に該当しない設計にする。

- ポイント有効期限：取得から2年（長期失効しない設計）
- 退会時のポイントは消滅
- 規約への明示が必須

### 8.4 不正利用対策
- 自分の図案を自分で買う行為の検出
- 同一IP・同一決済情報の異常検知
- 1日あたりの感謝ポイント贈呈上限

## 9. 収益モデル

### 9.1 図案販売手数料
- 販売額の25-30%（Etsyの取引手数料6.5%＋上場手数料より低めに設定）
- 残り70-75%のうち、作者への現金支払い分とポイント還元分に分割

### 9.2 サブスクリプション（プレミアム会員）
- **無料層**
  - 図案変換：月3回
  - 購入：無制限
  - 公開：3図案まで
- **有料層**：月額1,200円 or 年額12,000円
  - 図案変換：無制限
  - 高度な編集機能
  - 公開図案：無制限
  - 販売手数料割引（30%→15%）
  - 制作ログ無制限保管
  - 優先サポート

### 9.3 物販オプション（Phase 4以降）
- 完成作品の額装サービス
- 糸キット販売（指定図案に必要な糸セット）
- 完成作品の冊子化サービス

### 9.4 収益試算（Phase 4時点想定）
- アクティブユーザー：1万人
- 有料会員転換率：5%＝500人 × 1,200円 = 月60万円
- マーケットプレイス手数料：月100-200万円
- **月商目標：200-300万円**

## 10. 法務・著作権対応

### 10.1 ユーザー利用規約
- 写真は本人が権利を有するもののみアップロード可
- 第三者の肖像・キャラクター・ブランドロゴは禁止
- AIで自動検出し警告
- 違反時のアカウント停止条件を明示

### 10.2 著作権侵害対応
- DMCA準拠の通報フロー（米国向け）
- 24-48時間以内の対応コミット
- EU DSA（デジタルサービス法）対応
- 日本のプロバイダ責任制限法対応

### 10.3 AI学習データの透明性
- 既存市販図案を学習に使用しない
- パブリックドメイン画像＋ユーザー許諾済みデータのみ
- 透明性レポートを定期公開（差別化要素にもなる）

### 10.4 糸ブランドの扱い
- 色番号は事実情報として参照可能
- ロゴ・商標は使用しない
- 「DMC #310相当」のような中立表現を統一
- 必要に応じて公式ライセンス交渉

### 10.5 ポイント関連
- アプリ内のみ流通、現金換金不可で資金決済法対象外を維持
- 有効期限を明示
- 退会時の扱いを規約に明記

### 10.6 必要な法務アクション
- Phase 1着手前：弁護士スポット相談（写真→図案の権利関係）
- Phase 4着手前：マーケットプレイス運営の法務体制構築
- 各フェーズ前：利用規約・プライバシーポリシーの更新

## 11. 開発ロードマップ

### Phase 0：仮説検証（0-2ヶ月）
**目的**：作るべきか判断する

- Web上の単純フォーム or LINE公式アカウントで「写真送る→PDF返す」を手動運用
- 妻のネットワーク50-100人でテスト
- 検証項目：
  - 需要は実在するか
  - 価格感（300円？500円？1,000円？）
  - 既存サービスへの満足度
  - 何が一番嬉しいか
- **Go/No-Go判断**

### Phase 1：PWA MVP（2-6ヶ月）
**目的**：核心機能を市場に出す

- 写真→図案変換（DMC + Olympus + Cosmo）
- 多言語UI（日本語・英語）
- PDF出力 + ブラウザビューアー
- 都度課金（1図案500円）+ 簡易サブスク
- ユーザーアカウント・履歴管理
- **マーケットプレイス機能はまだ載せない**
- **このフェーズで成功すれば次に進む**

### Phase 2：制作支援強化（6-10ヶ月）
**目的**：使う頻度を増やす

- ブラウザビューアーで進捗管理
- 制作メモ機能（位置紐づけ）
- 完成記録・写真アップロード
- カテゴリ・タグ整理
- **ここまでPWAのみ**

### Phase 3：iOS / iPadOSネイティブ（10-15ヶ月）
**目的**：深い体験を提供

- SwiftUIで実装
- Apple Pencilで進捗マーキング・図案編集
- Core MLでローカル変換（AIコスト削減）
- オフライン対応
- App Store公開

### Phase 4：マーケットプレイス・ポイント経済（15-20ヶ月）
**目的**：プラットフォーム化

- 図案の公開・販売
- ポイント経済圏稼働
- 展示室機能
- 感謝ポイント贈呈
- 作者プロフィールページ
- **ここで初めて「堀」が完成**

### Phase 5：スケール展開（20ヶ月以降）
- macOSネイティブ
- ドイツ語・フランス語・ロシア語追加
- 物販オプション
- B2B（手芸店との連携）
- 上級者向け図案編集ツール

## 12. 開発体制とClaude Code活用

### 12.1 体制
- **設計・ビジョン・確認**：あなた（プロダクトオーナー兼アーキテクト）
- **コーディング実装**：Claude Code
- **テスト・UX確認**：あなた + 妻 + コアユーザー

### 12.2 Claude Codeとの協働原則

**1. CLAUDE.mdで文脈を維持**

リポジトリ直下に`CLAUDE.md`を置き、以下を常時更新：
- プロジェクトのビジョン（この仕様書の要約）
- 現在のフェーズ
- アーキテクチャ原則
- 命名規約・コーディングスタイル
- やってはいけないこと（外部API直接呼び出し禁止、特定ライブラリ使用禁止など）

**2. 機能を atomic タスクに分解**

1タスク=1機能=1PRを基本とする：
- 「写真アップロードAPIを作る」
- 「Stripe Webhookを実装する」
- 「展示室ページのレイアウトを作る」

各タスクで明確な受け入れ基準を提示。

**3. 型・契約から先に決める**

データモデル・API契約をあなたが先に確定→Claude Codeに実装させる。後から構造を変えるとリファクタが大きくなる。

**4. テスト駆動を活用**

重要なロジック（ポイント計算、画像変換、課金）はテストを先に書かせる。

**5. レビューサイクル**

- 小さい変更：差分確認のみ
- 大きい変更：実際に動かして確認
- 重要な変更：あなたが読んで質問する

**6. 危険操作の明示**

DB マイグレーション、外部API課金、本番デプロイは必ずあなたの確認を経る運用にする。

### 12.3 推奨リポジトリ構造
```
stitchlog/
├─ apps/
│  ├─ web/          # PWA (Next.js)
│  ├─ ios/          # SwiftUI
│  └─ admin/        # 管理画面
├─ packages/
│  ├─ api/          # 共通API
│  ├─ db/           # データベーススキーマ
│  ├─ types/        # 共通型定義
│  └─ utils/        # 共通ユーティリティ
├─ CLAUDE.md
├─ SPEC.md（この仕様書）
└─ README.md
```

モノレポでPWAとネイティブが型を共有することで、API変更時に両側の修正漏れを防ぐ。

### 12.4 ワークセッションの例

**良い指示例：**
> 「Patternエンティティに対応するPOST /api/patterns/createエンドポイントを作って。リクエスト：multipart/form-data（画像ファイル + JSON metadata）。バリデーション：画像は10MB以内、jpeg/png/heic。レスポンスは作成されたPatternのid。テストを先に書いて。」

**避けるべき指示例：**
> 「図案作成機能を作って」（範囲が広すぎる）

### 12.5 セッションごとのプロトコル

1. セッション開始時：`CLAUDE.md`と関連コードをClaude Codeに読み込ませる
2. タスクを1つだけ提示
3. 実装→テスト→レビュー
4. 完了したら`CLAUDE.md`を更新（変更点、決定事項）
5. コミット＆プッシュ

## 13. リスクと対処

| リスク | 影響度 | 対処 |
|--------|--------|------|
| 変換品質が既存ツールを超えられない | 高 | Phase 0で徹底検証、ダメなら撤退 |

| App Store審査リジェクト | 中 | PWAで先行リリース、ネイティブは後追い |

## 付録：このプロジェクトの位置づけ

| 既存案 | 立ち位置 |
|--------|---------|
| Letters to Me | 自己内省層向け、時間効果 |
| Receipt Stories | 文学化層向け、変換軸 |
| Echoes from Yourself | 自己対話層向け、時間×データ蓄積 |
| **Stitchlog** | **趣味コミュニティ層向け、ネットワーク効果＋時間効果のハイブリッド** |

Stitchlogの独自性は「時間が経つほど価値が増す」だけでなく**「ユーザーが増えるほど価値が増す」**ネットワーク効果を持つこと。他のプロジェクトとは異なるスケール上限を持つ。

他プロジェクトとの併走戦略として：
- Letters to Me：個人完結型、収益安定、ローンチが軽い
- Stitchlog：プラットフォーム型、長期成長、初期投資が大きい

両者は補完関係にあり、Letters to Meで収益基盤を作りながらStitchlogで成長を狙う構造が現実的。
