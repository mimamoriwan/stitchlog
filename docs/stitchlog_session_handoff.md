## 1. プロジェクトの出発点

### きっかけ
- 妻（クロスステッチ歴30年）の実体験からアイデアが生まれた
- 市販図案の種類の少なさ、著作権の壁、愛犬など個人的なモチーフを刺せない問題
- 完成作品の発表場所がSNSしかなく、制作知識が個人の中に埋もれる問題

### 妻からのヒアリングで判明した本質的なニーズ
- 愛犬の写真などをクロスステッチできたらステッチ好きには刺さる
- 出来上がった作品を見てもらう場がSNSしかない
- 制作中の気づき（注意点）を書き留める機能が自分の経験値になる
- 自分がデザインした図案に「ここが注意ポイント」がつけられたら新しいネットワークになりそう

## 2. 既存サービスの調査結果

### 競合として存在するもの
| サービス | 内容 | 弱点 |
|---------|------|------|
| Love It Stitch It | 写真→図案変換＋マーケット | 制作メモ・展示室・ポイント経済なし |
| Stitchly | iOS/Android変換＋進捗管理 | コミュニティ・マーケットなし |
| Xstitchify | 変換特化（コンフェッティ対策あり） | コミュニティゼロ |
| Etsy | 図案販売 | クロスステッチ特化UXなし |
| Pic2Pat / MyPhotoStitch | 無料変換ツール | 一方通行、品質が限界 |

### なぜ先行者がいるのに定着しないか
1. **変換品質が期待を超えていない**（技術的タイミングの問題→今が変わる）
2. **ツールがバラバラで統合されていない**（変換・販売・記録・共有が全部別サービス）
3. **作者と刺す人が繋がれない**（一方通行で終わる）
4. **日本語・日本糸ブランド対応がほぼゼロ**（妻が知らなかった根本原因）

## 3. アプリ概念「Stitchlog」の確定ビジョン

### 一言で
「クロスステッチ版のnote + Etsy + 進捗管理アプリ」を一つに統合したプラットフォーム

### コアバリュー
**「作る人と刺す人が固定されない経済圏」**
全ユーザーが図案作者にも刺し手にもなれる。

### 具体的なユーザー体験（シーン）

**シーン1：図案作成**
愛犬の写真をアップ→糸ブランド・色数・サイズを選択→アプリが図案を生成→購入

**シーン2：刺しながら使う**
iPadで図案を表示しながら進捗をタップマーキング→「13列目：色替え注意」などメモを追加→中断後に再開してもどこから刺すかわかる

**シーン3：完成・公開**
完成写真と図案・制作メモをセットで投稿→「注意ポイント付き」でプラットフォームに公開

**シーン4：別のユーザーが購入**
世界のユーザーが発見→購入→制作メモを読みながら刺す→完成したら作者に通知→作者の展示室に完成作品が追加される

### ポイント経済圏（確定）
```
図案が売れる → 作者に制作ポイント還元（売上の30%相当）
完成した人 → 作者に感謝ポイント贈呈
ポイントは他の図案購入に使える
現金換金は不可（資金決済法回避）
```

### 展示室（確定）
- 図案ごとに世界中の購入者の完成作品が集まる
- 国旗が並ぶことで「世界中で刺されている」が一目でわかる
- 作者の実績と、次に買う人への信頼証明を兼ねる

## 4. 変換品質問題と重大な気づき

### なぜ既存ツールの変換はぼやけるのか

**本質：これはAIの精度問題ではなく、設計思想の問題**

```
既存ツールの誤解
写真 → 色を近似 → グリッドに色を埋める → 完成

クロスステッチの実際
写真 → 何をどのステッチで表現するか判断 → 複数レイヤーで構成 → 完成
```

### 正しいパターンデータ構造
```
グリッドレイヤー（クロスステッチ本体）
バックステッチレイヤー（輪郭・細い線・ヒゲ）← 既存ツールが無視している
フレンチノットレイヤー（点・目の光）
ハーフ/クォーターステッチ（境界の滑らかさ）
```

### 犬猫のヒゲが潰れる理由
ヒゲは**バックステッチ**で表現するのが正解。グリッドに色を埋めようとすると消える。これは精度の問題ではなく**表現手段の選択ミス**という設計レベルの問題。

## 5. 変換フローの確定方針

### 質問：ChatGPT image2.0経由 vs 元写真直接

**結論：元写真を直接アプリに取り込む方法が正解**

理由：ChatGPT image2.0の「クロスステッチ風」はあくまで視覚的な模倣であり、クロスステッチ技法の知識がない状態で変換する。この段階でヒゲ等の細い要素が既に失われてしまい、その後スキルがあっても回復不可能。

```
❌ Aルート
元写真 → ChatGPT（ヒゲ消滅）→ アプリ → 図案（回復不可）

✅ Bルート
元写真（情報最大）→ スキルを持つアプリが直接判断 → 多層構造の図案
```

ChatGPT画像生成は**内部処理の補助**として使う余地はあるが、ユーザーの入口は常に「元写真1枚」に統一する。

## 6. 今回の最大の方針転換：スキル戦略

### Vercel skills.sh エコシステムとは
- Claude Codeなど18以上のAIエージェントで使えるスキルのパッケージマネージャー
- `npx skills add vercel-labs/agent-skills` のような形でインストール
- SKILL.mdファイルで構成されたモジュールをエージェントが必要時にロード
- skills.shがスキルのディレクトリ兼リーダーボードとして機能

### Stitchlog開発に使えるVercel公式スキル
| スキル | 用途 | インストール |
|--------|------|------------|
| vercel-react-best-practices | PWAのReact品質（57ルール） | `npx skills add vercel-labs/agent-skills` |
| web-design-guidelines | アクセシビリティ・UX（100+ルール） | 同上 |
| vercel-composition-patterns | コンポーネント設計パターン | 同上 |

### 最大の気づき：最重要スキルは存在しない→自分たちで作る

```
既存スキル：インフラ・UI品質のための汎用スキル
不在のスキル：クロスステッチ変換の専門知識スキル ← 誰も作っていない
```

クロスステッチは1000年近い歴史を持ち、技法体系は膨大な文献に記述されている。これはAIが最も得意とする領域。エンジニアが作ってきた既存ツールが変換品質で限界を迎えている根本理由は、**ドメイン知識がコードに組み込まれていないこと**。

### スキルを自作する戦略的価値

```
① クロスステッチ変換スキル（SKILL.md）を自社開発
② Stitchlogの変換エンジンで使用 → 品質で競合に差をつける
③ skills.shにオープンソースとして公開
④ スキルのインストール数が増える = Stitchlogの認知が広まる
```

スキル公開がそのままマーケティングになる構造。

### 奥様の役割の定義
- **知識の提供者ではなくバリデーター**
- AIが技法知識をスキル化 → 奥様が実際に刺して「これは正しい」「おかしい」を判定
- 30年の経験が品質保証として機能

## 9. 次セッションで継続すべき論点

以下が未解決のまま次に持ち越す主要な論点です。

### 論点1：クロスステッチ変換スキルの設計【最優先】

スキルとして言語化すべき知識体系の構築。

```
必要な知識マップ（案）
├─ 視覚要素 → ステッチ種別の変換ルール
│   ├─ 細い線・輪郭 → バックステッチ
│   ├─ 境界の滑らかさ → クォーターステッチ
│   ├─ 点・目の光 → フレンチノット
│   └─ テクスチャー → ロングアームクロスステッチ等
├─ 動物固有の課題と解法
│   ├─ ヒゲの表現
│   ├─ 毛並みのグラデーション
│   └─ 目の立体感
├─ 写真の「意訳」原則
│   ├─ 何を省くか
│   ├─ 色数の決め方
│   └─ 解像度とステッチ数のトレードオフ
└─ 糸ブランド別の色特性
```

→ **次セッションでClaude Codeが参照できるSKILL.mdとして構造化する**

### 論点2：変換エンジンのアーキテクチャ設計

スキルを変換パイプラインに組み込む具体的な実装方針。

```
元写真
  ↓
画像解析（細線・点・輪郭・面を識別）  ← どの技術を使うか
  ↓
スキルを参照して各要素のステッチ種別を決定  ← どう組み込むか
  ↓
グリッド + バックステッチ + フレンチノット の複合出力  ← データ構造
  ↓
PDFに全レイヤーが正確に印刷される  ← PDF仕様
```

### 論点3：Phase 0の具体的な実行計画

- LINE公式アカウント vs Webフォーム、どちらで始めるか
- 手動運用のフロー設計（写真受信→変換→PDF返送）
- 妻のネットワーク50-100人へのアプローチ方法
- 検証期間と判断基準（Go/No-Go）
